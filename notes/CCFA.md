## Domain 1 Agentic Architecture & Orchestration

### 1.1 エージェンティックループ

- stop_reasonの`tool_use`と`end_turn`の違い
- アンチパターン
  - 自然言語で停止のシグナルを記述する
  - 繰り返し回数を制限する
  - AIの応答から判断する

### 1.2 マルチエージェント

- Hub-and-spokeアーキテクチャ
- コーディネータのタスク分割が狭すぎるとカバレッジの不完全につながる
- カバレッジを満たすまでコーディネータはサブエージェントに再委任するようにする
- コミュニケーションはコーディネータを通す

### 1.3 サブエージェント設定

- コーディネータがサブエージェントを起動するにはallowedToolsに`Task`が含まれている必要がある
- 共通の話題からフォークしたい場合`/fork`コマンドで分岐できる
- コンテキストとメタデータを区別するために構造化されたフォーマットを使用する
- 複数のTaskツール呼び出しをまとめて呼び出しサブエージェントを並列に起動する

### 1.4 マルチステップワークフロー

- 絶対に守らせたいルールはプロンプトではなく仕組みで保証する
  - Stateで管理し各ツールが実行前に前提条件をチェック
  - デコレータで共通化する
  - State Machineで制御
- リクエストに複数の要素が含まれている場合は、まず同じコンテキストを使って並列で個別に調査し、結果を合成する
- エスカレーションする場合は構造化されたサマリーを渡す

### 1.5 Agent SDKフック

- `PostToolUse`でツールの結果を変換する（Unix timeなど）
- ツールを呼ぶ前にルールチェックし、違反していた場合は人間へのエスカレーションなど別のワークフローに遷移
- フックは決定的でプロンプトは確率的

### 1.6 タスクの分割

- 大きなタスクを一回でやらせず、小さなステップに分割して順番に実行し、前のステップの結果を次のステップに渡す
- 決まり切った作業ならプロンプトチェイニング、探索作業なら動的分解
- プロンプトチェイニングは始めにステップを固定する
- 動的分解はClaudeが実行中にステップを決める
- 大きなコードレビューをファイルごとの小さなレビューに分けてさらに別のファイル間整合性チェックを設ける

### 1.7 セッション管理

- セッション再開するとき、エージェントに前回のセッションからの変更を教えることが重要
- 古いツール結果を保持したセッションを再開するより、構造化したサマリーを渡して新しいセッションを始める方が良い
- 同じコードベース分析から2つのテスト戦略やリファクタを試したりする場合にフォークセッションが有効

## Domain 2 Tool Design & MCP Integration

### 2.1 ツールインタフェース設計

- ツールの説明はエージェントがツール選定するための主要な情報
- インプットフォーマット、クエリ例、エッジケース、境界説明をツール説明に書くことが重要
- エージェントが特定のキーワードに引きずられて誤ったツールを起動しないようなプロンプトをすることが必要
- 似ているツールは明確に目的や期待する入力、出力などを差別化し、似たツールと比較していつ使うのか説明する
- ツール名は他のツールと機能の重複がないような命名にする
- 汎用ツールを目的特化のツールに分割する
- システムプロンプトがツールの説明を上書きしないかどうかレビューする

### 2.2 MCPツールのエラーレスポンス

- 構造化されたエラーのメタデータを返すようにする
- エラーの種類
  - transient error：タイムアウトやサービス不能
  - validation error：不正な入力
  - business error：ポリシー違反
  - permission error
- 統一的で曖昧なエラーレスポンス（失敗しましたなど）はエージェントが正しくリカバリーできない
- もう一度実行すれば成功するかもしれないエラーと、何回やってもだめなエラーがある
  - レスポンスにリトライするべきか？という情報を構造化して含めることで無駄なリトライを防ぐ
- サブエージェントは自力でエラー解決を試み、解決できないものを結果と試行内容とともにコーディネータに渡す
- エラーと正しい空の結果を区別できるようにする

### 2.3 エージェントへのツール分配

- 1エージェントにツールを渡しすぎると正しい選択ができなくなる
- 自分の専門以外のツールを渡されると誤使用が多くなる
- `tool_choice`はClaude APIを呼ぶときにリクエストに含めるパラメータで、Claudeがツールをどう選ぶかの判断の自由度
  - `auto`：Claudeに任せる
  - `any`：必ず何らかのツールを使わせるがどれを使うかはClaudeが選ぶ
  - `forced`：特定のツールを必ず使わせる
- 専門外ツールでも頻繁に使うツールはエージェントに持たせ、複雑なケースのみコーディネータを経由する
  - 何でもかんでも専門外のことをコーディネータ経由で聞いていたら非効率なため

### 2.4 MCPのワークフローへの統合

- MCPサーバのスコープ
  - プロジェクトレベル（.mcp.json）
  - ユーザレベル（./claude.json）
- `.mcp.json`の中に環境変数を埋め込むことができる
  - `${GITHUB_TOKEN}`など
- MCPサーバが持っているツールは接続時にすべてロードされる
- MCPのリソース機能を使ってどんなコンテンツがあるのかをエージェントに提示でき、エージェントが探す手間が省ける
- ビルトインツールではなくMCPサーバを使ってもらうためには、説明を明確に書く

### 2.5 ビルトインツールの使用

- Grep：コンテンツ検索
- Glob：ファイルパスのパターンマッチング
- Read/Write：ファイル操作
- Editが非ユニークなテキストマッチによって失敗した場合、Read+Writeをフォールバックとして使う
  - Editは特定のテキストを探して別のテキストに置き換えるという処理をする
  - Read/Writeはファイル全体を読んで内容を把握し、ファイル全体を書き直す
- 最初から全部ファイルを読むのではなくまずGrepしてエントリーポイントを探すなど徐々に理解していく
- ラッパーモジュールを使っている場合、エクスポートされた名前をまず特定してから各名前を検索するというステップを取る

## Domain 3 Claude Code Configuration & Workflows

### 3.1 CLAUDE.mdファイルの設定

- 設定の階層
  - ユーザレベル：`~/.claude/CLAUDE.md`
  - プロジェクトレベル：`.claude/CLAUDE.md` or root CLAUDE.md
  - ディレクトリレベル：サブディレクトリのCLAUDE.md
    - トピックに特化したルールファイルとして使う
- `@`を使って外部ファイルを参照させることができる

  ```md
  # プロジェクト設定

  このプロジェクトはxxxのWebアプリです。

  ## コーディング規約

  @docs/coding-style.md

  ## Gitワークフロー

  @docs/git-workflow.md

  ## テスト方針

  @docs/testing.md
  ```

- なんでもかんでもインポートせず、関連するファイルのみをインポートする
- 大きなCLAUDE.mdファイルをトピック特化の小さなルールに分割して`.claude/rules/`に保存する
- `/memory`コマンドを使うと今どのメモリファイル（CLAUDE.mdなど）が読み込まれているか確認する
  - セッションごとにClaudeの挙動が違うという問題の原因を突き止める
