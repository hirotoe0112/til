## Domain 1 Agentic Architecture & Orchestration

### 1.1 エージェンティックループ

- stop_reasonの`tool_use`と`end_turn`の違い
- アンチパターン
  - 自然言語で停止のシグナルを記述する
  - 繰り返し回数を制限する
  - AIの応答から判断する

### 1.2 マルチエージェント

- Hub-and-spokeアーキテクチャ
- コーディネータのタスク分割が狭すぎるとカバレッジの不完全につながる
- カバレッジを満たすまでコーディネータはサブエージェントに再委任するようにする
- コミュニケーションはコーディネータを通す

### 1.3 サブエージェント設定

- コーディネータがサブエージェントを起動するにはallowedToolsに`Task`が含まれている必要がある
- 共通の話題からフォークしたい場合`/fork`コマンドで分岐できる
- コンテキストとメタデータを区別するために構造化されたフォーマットを使用する
- 複数のTaskツール呼び出しをまとめて呼び出しサブエージェントを並列に起動する

### 1.4 マルチステップワークフロー

- 絶対に守らせたいルールはプロンプトではなく仕組みで保証する
  - Stateで管理し各ツールが実行前に前提条件をチェック
  - デコレータで共通化する
  - State Machineで制御
- リクエストに複数の要素が含まれている場合は、まず同じコンテキストを使って並列で個別に調査し、結果を合成する
- エスカレーションする場合は構造化されたサマリーを渡す

### 1.5 Agent SDKフック

- `PostToolUse`でツールの結果を変換する（Unix timeなど）
- ツールを呼ぶ前にルールチェックし、違反していた場合は人間へのエスカレーションなど別のワークフローに遷移
- フックは決定的でプロンプトは確率的

### 1.6 タスクの分割

- 大きなタスクを一回でやらせず、小さなステップに分割して順番に実行し、前のステップの結果を次のステップに渡す
- 決まり切った作業ならプロンプトチェイニング、探索作業なら動的分解
- プロンプトチェイニングは始めにステップを固定する
- 動的分解はClaudeが実行中にステップを決める
- 大きなコードレビューをファイルごとの小さなレビューに分けてさらに別のファイル間整合性チェックを設ける

### 1.7 セッション管理

- セッション再開するとき、エージェントに前回のセッションからの変更を教えることが重要
- 古いツール結果を保持したセッションを再開するより、構造化したサマリーを渡して新しいセッションを始める方が良い
- 同じコードベース分析から2つのテスト戦略やリファクタを試したりする場合にフォークセッションが有効

## Domain 2 Tool Design & MCP Integration

### 2.1 ツールインタフェース設計

- ツールの説明はエージェントがツール選定するための主要な情報
- インプットフォーマット、クエリ例、エッジケース、境界説明をツール説明に書くことが重要
- エージェントが特定のキーワードに引きずられて誤ったツールを起動しないようなプロンプトをすることが必要
- 似ているツールは明確に目的や期待する入力、出力などを差別化し、似たツールと比較していつ使うのか説明する
- ツール名は他のツールと機能の重複がないような命名にする
- 汎用ツールを目的特化のツールに分割する
- システムプロンプトがツールの説明を上書きしないかどうかレビューする

### 2.2 MCPツールのエラーレスポンス

- 構造化されたエラーのメタデータを返すようにする
- エラーの種類
  - transient error：タイムアウトやサービス不能
  - validation error：不正な入力
  - business error：ポリシー違反
  - permission error
- 統一的で曖昧なエラーレスポンス（失敗しましたなど）はエージェントが正しくリカバリーできない
- もう一度実行すれば成功するかもしれないエラーと、何回やってもだめなエラーがある
  - レスポンスにリトライするべきか？という情報を構造化して含めることで無駄なリトライを防ぐ
- サブエージェントは自力でエラー解決を試み、解決できないものを結果と試行内容とともにコーディネータに渡す
- エラーと正しい空の結果を区別できるようにする

### 2.3 エージェントへのツール分配

- 1エージェントにツールを渡しすぎると正しい選択ができなくなる
- 自分の専門以外のツールを渡されると誤使用が多くなる
- `tool_choice`はClaude APIを呼ぶときにリクエストに含めるパラメータで、Claudeがツールをどう選ぶかの判断の自由度
  - `auto`：Claudeに任せる
  - `any`：必ず何らかのツールを使わせるがどれを使うかはClaudeが選ぶ
  - `forced`：特定のツールを必ず使わせる
- 専門外ツールでも頻繁に使うツールはエージェントに持たせ、複雑なケースのみコーディネータを経由する
  - 何でもかんでも専門外のことをコーディネータ経由で聞いていたら非効率なため

### 2.4 MCPのワークフローへの統合

- MCPサーバのスコープ
  - プロジェクトレベル（.mcp.json）
  - ユーザレベル（./claude.json）
- `.mcp.json`の中に環境変数を埋め込むことができる
  - `${GITHUB_TOKEN}`など
- MCPサーバが持っているツールは接続時にすべてロードされる
- MCPのリソース機能を使ってどんなコンテンツがあるのかをエージェントに提示でき、エージェントが探す手間が省ける
- ビルトインツールではなくMCPサーバを使ってもらうためには、説明を明確に書く

### 2.5 ビルトインツールの使用

- Grep：コンテンツ検索
- Glob：ファイルパスのパターンマッチング
- Read/Write：ファイル操作
- Editが非ユニークなテキストマッチによって失敗した場合、Read+Writeをフォールバックとして使う
  - Editは特定のテキストを探して別のテキストに置き換えるという処理をする
  - Read/Writeはファイル全体を読んで内容を把握し、ファイル全体を書き直す
- 最初から全部ファイルを読むのではなくまずGrepしてエントリーポイントを探すなど徐々に理解していく
- ラッパーモジュールを使っている場合、エクスポートされた名前をまず特定してから各名前を検索するというステップを取る

## Domain 3 Claude Code Configuration & Workflows

### 3.1 CLAUDE.mdファイルの設定

- 設定の階層
  - ユーザレベル：`~/.claude/CLAUDE.md`
  - プロジェクトレベル：`.claude/CLAUDE.md` or root CLAUDE.md
  - ディレクトリレベル：サブディレクトリのCLAUDE.md
    - トピックに特化したルールファイルとして使う
- `@`を使って外部ファイルを参照させることができる

  ```md
  # プロジェクト設定

  このプロジェクトはxxxのWebアプリです。

  ## コーディング規約

  @docs/coding-style.md

  ## Gitワークフロー

  @docs/git-workflow.md

  ## テスト方針

  @docs/testing.md
  ```

- なんでもかんでもインポートせず、関連するファイルのみをインポートする
- 大きなCLAUDE.mdファイルをトピック特化の小さなルールに分割して`.claude/rules/`に保存する
- `/memory`コマンドを使うと今どのメモリファイル（CLAUDE.mdなど）が読み込まれているか確認する
  - 期待するファイルが読み込まれているか確認することで、セッションごとにClaudeの挙動が違うという問題の原因を突き止める

### 3.2 カスタムスラッシュコマンド

- コマンドの階層
  - プロジェクトレベル：`.claude/commands/`
  - ユーザレベル：`~/.claude/commands/`
- Skillsはフロントマターでの設定の記述ができる
  - `context:fork`：スキルを別のコンテキストで実行する
  - allowed-tools：スキルが使えるツール
  - argument-hint：スキルに渡す入力例やヒント
- Skillsはオンデマンドでタスク特化、CLAUDE.mdは常に読み込まれ適用される

### 3.3 パス特化ルール

- `.claude/rules/`ファイルのフロントマターに`paths`でファイルのマッチングパターンを指定すると、マッチするファイルを編集するときのみルールが有効になる

  ```md
  ---
  paths: ["terraform/**/*"]
  ---

  # Terraform規約

  hogehoge
  ```

- ディレクトリレベルのCLAUDE.mdよりも良い点は、コードベースに散らばったファイルをパターンで指定できること

### 3.4 プランモードと直接実行

- プランモードは複雑なタスクや大規模な変更、複数のアプローチがある場合、アーキテクチャ設計、複数ファイルの変更といった場合に使用する
- 直接実行はシンプルでスコープが明確なタスクに使用する
- 探索サブエージェントを使い結果を要約してメインに返すことで、メインのコンテキストを汚さずに済む
- プランモードで調査し、直接実行で実装する

### 3.5 反復的な改善テクニック

- 文章での説明だと解釈や結果が不整合になる場合、具体的な入力、出力の例を示すのが効果的
- 最初に期待する結果やエッジケースを含むテストを書き、テストの失敗を元に反復的に改善していく
- 実装を始める前にClaudeに質問させることで、開発者が気づいていなかった考慮事項を表面化できる
- 複数の問題が絡み合っている場合は、まとめて1メッセージで渡すことで、全体の整合性が保てる
- 独立した問題が複数ある場合は、一つずつ順番に渡すことで各修正に集中できる

### 3.6 CI/CDへの組み込み

- `-p`フラグ：Claude Codeをノンインタラクティブモードで動かす
- `--output-format json`：出力形式をjsonにする
- `--json-schema`：jsonスキーマ（どんな項目や型があるかなど）を指定する
- CLAUDE.mdはCI上でのClaude Codeでも同じように働く
- コードを生成したセッションと同じセッションでレビューさせるとバイアスが入り効果的ではない
- リランする際、コンテキストに前のレビューでの指摘事項を含めることで重複した指摘を防ぐ
- 重複したテストを作らせないために既存のテストを提供する
- テストの標準などをCLAUDE.mdに記載することでテスト生成の品質が上がる

## Domain 4 Prompt Engineering & Structured Output

### 4.1 プロンプト設計

- 曖昧な指示ではなく明確な基準を示す
  - 例）× 正確かどうかチェックして　〇 挙動がコードの挙動と矛盾する場合のみコメントして
- 自信度ベースのフィルタリングではなく明確なレビューの基準を示す
- false positiveが多いと本当に当たっていても開発者からの信頼が低下する
  - false positiveが多いカテゴリーは一時的にオフにして開発者からの信頼を回復しつつ、プロンプトを改善する
- 一貫性のある分類をするため、コード例とともに明確な重大度の基準を定義する

### 4.2 few-shotプロンプト

- 一貫性のない結果に対して、一貫性のあるフォーマットにするための最も効果的な手法
- 曖昧なシナリオに対して、なぜこのアクションが他の選択肢を差し置いて選ばれるのか？を示すのにも有効
- 新しいパターンに対して判断を下すのにも有効
- データ抽出のタスクにおいてハルシネーションを軽減するのにも有効
- 本当の問題と許容なコードを見分け、false positiveを減らすのにも有効
- 多様な構造の文書を扱うときにも有効
  - この構造の文書からはこう抽出するというお手本を示す
  - 必須項目が空になる問題を解決

### 4.3 構造化出力を保証する

- スキーマに準拠した構造化出力を保証するには`tool_use`とJSONスキーマが最も信頼性が高い
  - ツールを呼ぶと見せかけて実際には呼ばず、`tool_use`のレスポンスからツールの入力となるJSONだけをもらう
- toolの使用を介したJSONスキーマは構造エラーは防げるが、セマンティックエラーは防げない
  - 例）値が間違った項目に入っている
- enum形式にはよくわからないケースのための`unclear`とカテゴリにないデータのための`other`を追加する
- 元の文書が値を含まない可能性がある場合、スキーマの項目をoptionalにしておくことで、必須項目を満たすために虚偽の値で埋めるのを防げる
- 厳格な出力スキーマとともにフォーマットの正規化ルールをプロンプトに含めることで、ソースの一貫性のないフォーマットに対処する

### 4.4 抽出の品質を改善する

- リトライする際に正しく修正できるように、具体的なバリデーションエラーをプロンプトに付け加える
- 元のドキュメントに必要な情報がない場合などはリトライしても意味がない
- フォーマットエラーなどはリトライが有効
- どんなコードが指摘を引き起こしたかを記録することで、false positiveのパターンを体系的に分析できる
- セマンティックエラーはスキーマの強制では防げないが、スキーマ設計を工夫することで検出させることができる
  - 例）文書に記載された合計とClaudeが自分で計算した合計を含め、矛盾を検出する

### 4.5 バッチ処理

- Message Batch APIは50%のコスト節約、24時間までの処理時間
- ノンブロッキングで遅れても大丈夫なワークロードに適しているが、ブロッキング処理には向いていない
- 一つのリクエストに対するマルチターン呼び出しはサポートしていない
- すぐ答えが返ってこないのでどのリクエストのレスポンスか分からなくなるため、`custom_id`フィールドを使ってリクエストとレスポンスを紐づけることができる
- 失敗したドキュメントのみ必要な変更を加えてリトライする
- 大きなバッチを実行する前にサンプルセットを使ってプロンプトを改善し、成功率を高めリトライを減らす

### 4.6 マルチインスタンスレビュー

- セルフレビューは自分の決定に対してバイアスがあるため、コードを生成したClaudeとは独立したClaudeインスタンスを使ってレビューする
- 大きなレビューをファイルごとのステップに分け、コード全体のチェックも入れることで、注意が分散するのを防ぐ
- レビュー時に自信度を報告させ、自信度に応じてレビューのルーティングを決定する

## Domain 5 Context Management & Reliability

### 5.1 コンテキスト管理

- モデルは始めと終わりの情報は正確に処理するが中間部分の情報を省く傾向がある
- ツールの結果はコンテキストに蓄積してトークンを消費する
  - 必要な情報のみトリミングすることが重要
- 会話の一貫性を保つために完全な会話履歴を後続のAPIに渡すことが重要
- 正確さが絶対に必要な情報は、要約される会話履歴とは別に毎回のプロンプトに含める
- 位置の影響を軽減するために重要な情報を集約された情報の最初に置き、詳細な情報は明示的なセクションヘッダーを使って整理する
- 構造化データにメタデータを含めるようにサブエージェントに要求し、後続の処理の正確性を確保する
- 後続のエージェントにコンテキストの余裕がない場合、上流のエージェントは、長大なコンテンツではなく重要な事実を含む構造化データを返すようにする

### 5.2 効果的なエスカレーション
