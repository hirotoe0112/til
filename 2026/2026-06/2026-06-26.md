## DSQL

- Prisma用のツール
  - https://github.com/awslabs/aurora-dsql-orms/tree/main/node/prisma

以下のマイグレーションファイルが存在するプロジェクトがあるとする

```
model users {
  id         Int       @id @default(autoincrement())
  first_name String    @db.VarChar(100)
  email      String    @unique @db.VarChar(255)
  created_at DateTime? @default(now()) @db.Timestamp(6)
  age        Int?

  posts posts[]
}

model posts {
  id         Int       @id @default(autoincrement())
  author     users     @relation(fields: [authorId], references: [id])
  authorId   Int
  created_at DateTime? @default(now()) @db.Timestamp(6)

  @@index([created_at])
}
```

```sql
-- CreateTable
CREATE TABLE "public"."users" (
    "id" SERIAL NOT NULL,
    "name" VARCHAR(100) NOT NULL,
    "email" VARCHAR(255) NOT NULL,
    "created_at" TIMESTAMP(6) DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT "users_pkey" PRIMARY KEY ("id")
);

-- CreateIndex
CREATE UNIQUE INDEX "users_email_key" ON "public"."users"("email");
```

```sql
-- AlterTable
ALTER TABLE "public"."users" ADD COLUMN     "age" INTEGER;
```

```sql
ALTER TABLE "public"."users" DROP COLUMN "name",
ADD COLUMN     "first_name" VARCHAR(100) NOT NULL;
```

```sql
-- CreateTable
CREATE TABLE "posts" (
    "id" SERIAL NOT NULL,
    "authorId" INTEGER NOT NULL,
    "created_at" TIMESTAMP(6) DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT "posts_pkey" PRIMARY KEY ("id")
);

-- CreateIndex
CREATE INDEX "posts_created_at_idx" ON "posts"("created_at");

-- AddForeignKey
ALTER TABLE "posts" ADD CONSTRAINT "posts_authorId_fkey" FOREIGN KEY ("authorId") REFERENCES "users"("id") ON DELETE RESTRICT ON UPDATE CASCADE;
```

```
✗ Missing relationMode = "prisma" in datasource block (line 6)
  → Add relationMode = "prisma" to your datasource block. DSQL does not support foreign key constraints.
```

- DSQLは外部キーをサポートしていない
  - relationModeを`prisma`にする必要がある
  - relation modeとはレコード間の関係がどのように適用されるかを指定する
    - `foreignKeys`と`prisma`がある
    - [Prisma のリレーションモードを使用してレコード間の関係を管理する | Prisma ドキュメント](https://www.prisma.io/docs/orm/prisma-schema/data-model/relations/relation-mode)
    - デフォルトで`foreignKeys`となっている
    - `foreignKeys`モードは、外部キーを使用してリレーションが強制される
    - `prisma`モードはDBの制約ではなくPrismaによって処理される

## 質問の仕方

- 質問ができない=勇気が足りない=失敗が足りない
- 小さな失敗を積み重ねる
  - 小さな失敗がたくさんあれば少しぐらいの失敗はどうでも良い
  - 失敗を避けようとして成功でも失敗でもない選択肢を取ることが問題
- やった方がいいことはやる
- 仕事がうまくいくためなら性格を多少は変える
- 人から嫌われているか好かれているかはわからないので、そんな曖昧なもので判断を濁らせるのはNG

### 質問の型

| 型               | 目的               | 例文                                                 |
| ---------------- | ------------------ | ---------------------------------------------------- |
| 明確化           | 言葉の定義を揃える | 「〜とおっしゃるとき、具体的に何を指していますか？」 |
| 前提の可視化     | 隠れた仮定を出す   | 「それはXXを前提としていますか？」                   |
| 証拠の要求       | 根拠を問う         | 「それを支持するデータはありますか？」               |
| 因果の深掘り     | メカニズムを問う   | 「なぜそうなると思いますか？」                       |
| 選択肢の展開     | 思考を広げる       | 「他に考えられる選択肢はありますか？」               |
| 意思決定の明確化 | 判断基準を問う     | 「何があれば考えを変えますか？」                     |
| フォローアップ   | 表面で終わらせない | 「もう少し詳しく聞かせてもらえますか？」             |

実践法： 上記の一覧を1枚にまとめたカードを作り、重要な会議の前に手元に置く。

### ベースラインの認知

#### 会議後に自分の質問を記録し、ラベルを貼る

ゴール：「自分は誘導質問が多い」「フォローアップをしていない」などパターンを認識する。

- [ ] 明確化型
- [ ] 深掘り型
- [ ] リーディング（誘導）型 ← これが多い人は要注意
- [ ] クローズド型（Yes/Noで終わる）
- [ ] フォローアップ型

#### フォローアップ質問を鍛える

誰かが重要な発言をしたら、自分の意見を言う前に必ず2回フォローアップする。

- 1回目：明確化「〜とはどういう意味ですか？」
- 2回目：深掘り「それの主な原因は何だと思いますか？」

#### 会議録音・議事録レビューと書き直し

「あのとき聞けばよかった質問」を書き直す習慣。

1. 会議録音 or 議事録から5分を選ぶ
1. 質問のチャンスを逃した瞬間を3つ特定する
1. その瞬間の「理想の質問」を書き直す
1. そのフレーズを次の会議のカードに追加する

### なぜ失敗が怖いのか

- 勇気とは恐怖を感じながら行動する能力であり、恐怖がない状態のことではない
- × 質問して的外れだったら → 自分は頭が悪い人間だ
- 〇 質問して的外れだったら → その分野の知識がまだ足りない
- 小さなリスクの繰り返しが恐怖の閾値を下げる

| 段階 | 行動                                                 | 怖さのレベル |
| ---- | ---------------------------------------------------- | ------------ |
| ①    | 会議後に個別チャットで質問する                       | ★☆☆☆☆        |
| ②    | 発言が少ない小さなMTGで明確化の質問をする            | ★★☆☆☆        |
| ③    | 普段の会議で「もう少し詳しく聞かせてください」と言う | ★★★☆☆        |
| ④    | 前提を問う質問を上司のいる会議でする                 | ★★★★☆        |
| ⑤    | 多人数の場で意思決定に異議を唱える質問をする         | ★★★★★        |

①を十分に繰り返して「怖くなくなったら」②へ進む。

### 失敗の振り返り

自己否定せず、以下を考える。

1. 何が起きたか？（事実）
1. コントロールできた部分はどこか？
1. コントロールできなかった部分はどこか？
1. 失敗したのは「自分」か「戦略」か？
1. 次に試すことは何か？
1. 24時間以内にできる一歩は？

### 考え方のフレームワーク：「〜しない」を「〜する」に変換する

「禁止ルール」は行動を萎縮させる。「行動ルール」は次の成功を作る。

1. 「〜しない」という結論を書き出す
1. 「では、代わりに何をすれば良かったのか？」を問う
1. 「〜する」という行動に書き換える
1. 「その行動は、今後の成長に繋がるか？」を確認する

### やった結果、「何もしない」より悪化するもの

報告・連絡・相談の「形だけ版」

| 行動                                                          | なぜ有害か                                           |
| ------------------------------------------------------------- | ---------------------------------------------------- |
| 問題を報告するが代替案も謝罪もなく事実だけ投げる              | 聞いた側に全負荷がかかる。信頼でなく負担を渡している |
| 相談と称して愚痴・不満だけ言う                                | 相手の時間と感情エネルギーを消費。解決に近づかない   |
| 「ご確認ください」とだけ書いてファイルを送る                  | 何を確認してほしいか不明。判断コストを相手に丸投げ   |
| 問題発生後に事後報告で「一応報告しておきます」                | 対処不能なタイミングで情報を出すのは機能しない       |
| 遅延の報告なしに締切を過ぎる → 後から「言いませんでしたっけ」 | 最悪レベル。信頼が一撃で失墜する                     |

会議・議論での逆効果行動

| 行動                                                      | なぜ有害か                                         |
| --------------------------------------------------------- | -------------------------------------------------- |
| 根拠なく反対する（「なんかうまくいかない気がする」）      | 議論が止まる。感情的抵抗と区別がつかない           |
| 賛成だけして何も言わない × 会議後に「あれは問題だと思う」 | 意思決定プロセスを事後に汚染する                   |
| 前提確認なく自分の意見を長く話す                          | 論点がずれたまま会議時間を消費する                 |
| 決定済み事項を蒸し返す（合意後に「そもそも論」）          | 場の信頼とモメンタムを破壊する                     |
| 人の発言中に割り込んで自分の話に切り替える                | 発言者の意図が伝わらなくなる＋心理的安全性を下げる |
| 議事録なしに口頭合意を大量に積み上げる                    | 後で全員の認識がバラバラになることが確定している   |

依頼・指示・フィードバックの「雑版」

| 行動                                                                   | なぜ有害か                                       |
| ---------------------------------------------------------------------- | ------------------------------------------------ |
| 目的を伝えずに作業だけ依頼する                                         | 相手が最適な判断をできない。手戻りが発生する     |
| フィードバックを人格・性格への批判として行う（「あなたは詰めが甘い」） | 行動が改善されず、関係と心理的安全性が壊れる     |
| 褒めるが具体性ゼロ（「よかったよ」だけ）                               | 次回何を再現すべきかわからない。成長に寄与しない |
| 何度も同じことを指摘する（記録・仕組みなし）                           | 個人攻撃に変化し、相手が萎縮するだけ             |
| 「なんかおかしい」と言って修正箇所を特定しない                         | 受け取る側が何を直せばいいかわからない           |

コミュニケーションの「やってる感だけ版」

| 行動                                               | なぜ有害か                                           |
| -------------------------------------------------- | ---------------------------------------------------- |
| 話を聞いている間ずっとうなずくが何も言わない       | 理解・同意・懸念のどれかが全く判別できない           |
| 「わかりました」と言って動かない                   | 嘘の合意が形成される。後で「聞いてない」問題が発生   |
| 質問するが答えを聞く前に自分の意見を話す           | 情報収集でなく承認欲求の行動になっている             |
| メールのCCを大量につけてリスクヘッジする           | 全員の注意力を分散させ、誰も責任を持たない状態を作る |
| 「共有しておきます」だけで何のアクションも求めない | 受信者は何をすべきかわからない。情報の墓場になる     |

最も致命的な「信頼破壊行動」

| 行動                                             | なぜ致命的か                                       |
| ------------------------------------------------ | -------------------------------------------------- |
| ミスを隠す・矮小化して報告する                   | 後で発覚した時のダメージが指数関数的に増える       |
| 「言った・言わない」問題を口頭のみで処理し続ける | 信頼でなく記憶力の勝負になる。組織が機能不全に陥る |
| 他者の発言・成果を自分のものとして扱う           | 一度でも発覚すると全ての信頼が消える               |
| 相手のいない場で批判し、本人の前では何も言わない | 噂として必ず広がる。評判が確実に下がる             |
| 頼まれていないのに相手の仕事に介入・修正する     | 信頼でなく支配と受け取られる                       |

### 失敗の定義を変える

- × うまくできなかった⇒失敗
- 〇 行動しなかった⇒失敗、行動したけどうまくできなかった⇒成功
