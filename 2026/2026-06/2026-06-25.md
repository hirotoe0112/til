### 11:30-12:10 エージェンティック AI のためのデータ活用実践ガイド

- エージェンティックAI
  - reason（推論）、act（行動）、remember（記憶）
- ReActループ
  - 1回目のループ
    - 状態を推論してツール呼び出しして結果を保存
  - 2回目のループ
    - 状態を見て別のツールを呼び出して結果を保存
  - 3回目のループ
    - 状態を見てまた別のツールを呼び出して結果を保存
  - 状態が少しずつリッチになっていき次のアクションが決まる
  - 情報の保存容量（コンテキストウィンドウ）がある
- データはどこに保存されるのか
  - エージェントにはメモリが必要
  - 短期、長期それぞれ保管先が違う
- 既存のデータベースなどを捨てる必要はなくMCPでラップする
- AIデータ利用者の優先事項

  - データへのアクセス
    - IDを伝播する仕組みでアクセス権限を維持する
  - セキュリティ
    - きめ細かいアクセス制御
  - 低レイテンシー
    - Glue Materialized Views
- AIデータ生産者の優先事項

    - Glue Data Qualityでパイプライン内にルールを組み込む
  - データに意味を与える
    - カラムレベルのメタデータやタグ
  - 監査
    - 利用状況を追跡可能にする
- これから始めるべきこと
  - データをAPIでカプセル化する
  - APIをMCP化する

### 13:30-14:10 AI エージェントのオブザーバビリティ実践

- AIエージェントがどんな動きをするのか人間から見えづらくなる
- オブザーバビリティの基本
  - メトリクス
  - ログ
  - トレース
- 生成AIのオブザーバビリティの特徴
  - 正常に動作しているときにも能動的に観測する必要がある
  - 生成AIの失敗は「エラー」とならないことが多い
    - 応答は正常だが、内容がずれているなど
- 生成AIのオブザーバビリティの課題
  - 同じ入力でも毎回異なる動きをする
  - 複雑に連鎖する
  - 呼び出しの収集と分析をする必要がある
  - パフォーマンスだけでなく応答の品質も評価が必要
- OpenTelemetry
  - メトリクス、ログ、トレースを集めるためのオープンソースのフレームワーク
  - データの作成と送信について、標準仕様と実装
- AWS Distro for OpenTelemetry
  - AWS上でOpenTelemetryを使う場合
- CloudWatch生成AIオブザーバビリティ
  - Bedrock AgentCore、Lambda、EC2など実行環境に依存せず記録できる
  - AgentCore上で動かす場合は追加作業はほぼ不要
  - 主な機能
    - メトリクス
      - トークン使用量、レイテンシー、スロットリング、エラー等
      - Model invocationsダッシュボード
    - 機密データの保護
    - End-to-Endのトレース
- Amazon Bedrock AgentCore Evaluations
  - 応答の品質を評価する
  - LLM-as-a-judgeをマネージドで提供する
  - 採点結果はCloudWatchに集約
  - 情報への忠実さ、指示への忠実さ、有用性、回答の的確さを評価

### 14:30-15:10 実践！Amazon RDS と Amazon Aurora のコスト最適化とパフォーマンス向上

- パフォーマンス向上とコスト最適化は両立できる
- Auroraのコスト要素
  - コンピュート、ストレージ
  - バックアップ、データ転送、IOPS
  - その他、マルチAZ等、要件に応じたコスト
- コストを検討する際は↑の上から検討するのがおすすめ
- 実行効率の悪いSQLによるCPU使用率急増
  - 用途に合わせたファミリー
    - tファミリー：バースト可能、Mファミリー：汎用、Rファミリー：メモリ最適化
    - CloudWatch Database Insights：サイロ化したメトリクスを統合して見れるようにしたダッシュボード
  - Database Insightsを使用し、SQLの最適化とインスタンスの適正化
- ノイジーネイバー
  - 他のワークロードが影響を与えている
  - 垂直スケーリングと水平スケーリングが考えられる
    - コストを考慮して選択
- RDSのストレージ
  - Amazon EBSを利用
  - gp2：IOPSとストレージ容量が比例、gp3：ストレージ容量と独立してIOPSとスループットを割り当て可能、Provisioned IOPS：ミッションクリティカル向け
  - Database Insightsを使用し、ストレージクラスの適正化
- レポート用クエリで一時ファイルの生成が多い
  - Database InsightsでIO:BufFileWriteイベントが多発
  - 一時書き込みが発生するのはメモリ内で処理できなかった場合（work_mem）
  - EBSより低遅延であるローカルのNVMeベースのSSDを使用する
    - Amazon RDS Optimized Reads
- Auroraのストレージ
  - 容量とRead/Writeに対して課金、自動スケール
  - I/O-Optimizedでコスト予測性が向上
- Auroraで読み取りレイテンシー
  - Aurora Optimized Reads for Aurora PostgreSQL
    - インスタンスの名前に"d"がついたタイプに切り替えるだけで利用可能
    - インデックス再構築も高速化が見込める
    - 階層型キャッシュによりクエリ性能向上
    - OLTPワークロードにも効果あり
- バックアップコストの削減
  - コスト効率の高い長期バックアップ
    - S3へのApache Parquet形式でのエクスポート
    - S3への論理バックアップ（pg_dumpなど）
    - ストレージクラスの最適化
  - 古いスナップショットをS3にオフロード
